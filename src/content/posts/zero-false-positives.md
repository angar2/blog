정부지원 공고를 감지해 알림을 보내는 운영 도구 gov-radar를 만들면서, 결정론 코드와 LLM 판단이 한 파이프라인에 같이 놓이는 지점을 만났다. 수집한 공고를 프로필과 대조해 적격만 추리는 과정에서, "이 대조를 어디까지 코드가 확정하고 어디서부터 판단을 넘겨야 하나"라는 질문에 부딪혔다.

> 결정론 필터는 **"적격이다"를 증명하지 않는다.** <u>명백히 부적격이라는 근거가 나올 때만</u> 거르고, 조금이라도 애매하면 무조건 통과시킨다.

이 원칙을 *zero false positives*(허위 적격 0)라 부른다. gov-radar의 `src/match.mjs`가 이 원칙을 코드로 옮긴 결과물이다.

## hardCheck — 세 갈래 모두 확신이 설 때만 거절한다

`match.mjs` 파일 헤더는 이 모듈의 책임 범위를 미리 좁혀둔다. "명확한 부적격을 코드가 걸러내는 것"까지만 책임진다는 선언이다. 소프트판정(자격 텍스트를 읽고 애매한 경우를 가려내는 LLM 판단)은 이 모듈에 없다 — 그건 뒷단(헤르메스, 크론 잡 에이전트)의 몫으로 코드 밖에 남겨뒀다.

그 경계선의 코드 쪽 마지막 관문이 `hardCheck`다.

```js title="src/match.mjs"
export function hardCheck(profile, f) {
  if (f.ageLmt && f.minAge && f.maxAge && (profile.age < f.minAge || profile.age > f.maxAge))
    return `연령 요건 만 ${f.minAge}~${f.maxAge}세 — 프로필 ${profile.age}세가 벗어남`;
  if (f.regionCodes.length && profile.region_code && !f.regionCodes.includes(profile.region_code))
    return `거주지역 한정 정책 — 프로필 지역(${profile.region} ${profile.region_code}) 미포함`;
  if (f.jobCds.length && !f.jobCds.includes("0013010") && profile.employment_code && !f.jobCds.includes(profile.employment_code))
    return `취업상태 요건 [${f.jobCds.map((c) => JOB_CODE[c] ?? c).join(", ")}] — 프로필(${JOB_CODE[profile.employment_code] ?? profile.employment_code}) 불포함`;
  return null;
}
```

세 체크 모두 조건 여러 개가 **동시에 참일 때만** 거절 문자열을 반환하는 연쇄 AND로 짜여 있다.

- **연령**: 연령 제한 플래그가 있고, 상·하한이 둘 다 존재하고, 프로필 나이가 그 범위를 실제로 벗어날 때만 거절한다. 상한이나 하한 중 하나라도 비어 있으면 이 줄은 아예 작동하지 않는다.
- **지역**: 지역 한정 코드 목록이 비어 있지 않고, 프로필에 지역 코드가 있고, 그 코드가 목록에 없을 때만 거절한다. 빈 목록은 "전국"으로 해석한다 — 정책에 지역 정보가 없으면 지역을 이유로 거절하지 않는다.
- **취업상태**: 목록이 비어 있지 않고, 그 목록에 "제한없음"을 뜻하는 코드(`0013010`)가 없고, 프로필에 재직코드가 있고, 그 코드가 목록에 없을 때만 거절한다. 목록에 "제한없음"이 하나라도 섞여 있으면 이 줄은 통째로 무력화된다.

데이터가 하나라도 비어 있거나 애매하면 — 나이 상한 미기재, 지역 코드 없음, 재직코드 미기재 — 어느 줄도 걸리지 않고 함수는 `null`을 반환한다. 마감일을 현행·만료로 가르는 `computeStatus` 역시 같은 결정론 층에서 "계산으로 결정 가능한 것은 코드가 확정한다"는 원칙을 공유하는 이웃 함수다.

여기서 `null`을 어떻게 읽느냐가 이 설계의 핵심이다. `null`은 "적격이다"가 아니다. **"코드가 보기엔 명확한 결격 사유를 못 찾았다"**는 뜻이다. 함수 주석도 이를 못박는다 — 명확한 부적격이면 사유 문자열, 아니면 `null`(→ 소프트판정 대상). 즉, `null`은 판정의 완료가 아니라 **판단 위임**의 신호다.

## 비유 — 공항 검색대는 통과를 무해 증명으로 쓰지 않는다

공항 검색대는 위험물이라는 증거가 나올 때만 막는다. 가방 안에 뭔가 애매한 형체가 잡히면 그 자리에서 무해하다고 단정하지 않고 정밀검사로 넘긴다. 검색대를 통과했다는 사실 자체가 "무해함을 증명"하지는 않는다 — 단지 "명백한 위험물을 발견하지 못했다"는 뜻일 뿐이다.

`hardCheck`가 정확히 이 구조다. 통과(`null`)는 적격 판정이 아니라 명백한 결격을 발견하지 못했다는 뜻이고, 애매한 case는 뒷단의 정밀검사(소프트판정)로 넘어간다.

## 실측이 확인한 비대칭 — 표본 만료 누수 0

이 설계가 실전에서 얼마나 걸러내는지는 R2 검수 실측으로 확인됐다. `--full` 모드 실행에서 온통청년 2,648건 중 만료 1,042건이 컷되고, 하드체크가 다시 1,373건을 컷해 최종 후보 233건이 남았다. 이 과정에서 표본 만료 누수는 0으로 확인됐다. 마이홈은 270건 중 서울 접수중 6건(유효 1건)만 남았다.

이 233건이 곧바로 "적격"으로 확정되는 건 아니다. 하드체크를 통과한 뒤에도 헤르메스가 다시 소프트판정을 하는데, 이때도 같은 원칙이 반복된다 — 불확실하면 무조건 `[확인 필요]` 태그를 붙여 발송한다. 실패 모드를 **허위 적격**(위험한 실패)이 아니라 **딱지 붙은 도착**(안전한 실패)으로 강제하는 안전장치다. 결정론 층에서 시작된 비대칭이 다음 층까지 끊기지 않고 이어지는 셈이다.

## 즉, 정직함은 통과를 다 허용하는 데 있다

즉, 이 설계의 정직함은 애매한 걸 다 통과시키는 관대함에 있는 게 아니라, <u>확신이 서지 않으면 거절하지 않는다</u>는 절제에 있다. `hardCheck`가 "허위 적격 0"을 달성하는 이유는 코드가 똑똑해서가 아니라, 코드가 자기 확신의 한계를 정확히 알고 그 밖을 다음 층으로 넘기기 때문이다.

이 재사용 원칙은 gov-radar에만 묶이지 않는다. 결정론 코드와 LLM 판단이 같이 놓이는 어떤 시스템이든, 경계를 "코드가 확신할 수 있는 것만 코드가 맡는다"로 그으면 실패 모드가 자동으로 안전한 쪽으로 기운다. 판단을 넘기는 게 실패가 아니라 설계라는 뜻이다.
